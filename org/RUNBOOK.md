# 需求流水线（枢衡堂）

主人给出一个**正式需求**时，按下列阶段执行；各职在对应 `SOUL.md` 中有摘要，细节以本文件为准。

## 决策简报（枢衡 → 主人，飞书）

正式推进前或发现高价值机会时，枢衡用 **一条消息** 尽量包含：

1. **机会**：一句话 + 价值假设（赚什么、省什么、规避什么）。  
2. **分析**：计府结论摘要（区间、关键假设、待补数据）。  
3. **方案**：绳墨检索链结论（复用 Skill/CLI/最小实现路径）+ 拟派执行岗。  
4. **风险**：司驳可预审点（资金/合规/不可逆操作由 **主人终裁**，智能体不擅自代下财务决定）。  
5. **下一步**：需主人只选 **Go / No-Go / 补信息** 之一即可继续。

对外回复（飞书）时遵守 **`org/DELIVERY-FAITH.md`**：不编造图片路径与授权链接；新群接入见 **`org/FEISHU-GROUPS.md`**。

## 阶段 1 — 拆解（计府为主，枢衡协同）

- **计府**：把需求拆成可交付项、里程碑、依赖；写清范围边界。
- **枢衡**：与主人对齐意图，把旨意转给计府；必要时补充上下文。

## 阶段 2 — 市场 / 价值 / 成本（计府）

- **计府**：回答「有没有市场、有没有价值、成本多少」——一律 **区间与假设**，标明不确定项；**主动点出信息差或套利/降本空间**（若有），无依据不装精确。
- 若信息不足，列出 **待补数据**，由主人或枢衡决定是否继续。

## 阶段 3 — 任务分配（枢衡为主，计府输入）

- **枢衡**：根据计府拆解与成本区间，把子任务派给 **匠作 / 丹青 / 镜语**（或其一），在任务单或对话中写清 **验收标准** 与 **成本敏感点**（例如少轮次、小上下文）。
- **计府**：可提供「建议派谁、为何」的备注，不代替枢衡拍板。

## 阶段 4 — 能力缺口（绳墨 → 主人审批）

- **绳墨**（每次正式需求、指派执行前必须走一遍，**优先用工具省 token**）：
  1. **社区与技能**：用 `skills/find-skills` 指引 + `npx clawhub search …`（或 OpenClaw Directory / GitHub）查是否已有 **同款 Skill、参考实现、示例仓库**；本机 `workspace/skills/` 是否已覆盖。
  2. **现成 CLI / 二进制**：若无 Skill，查领域常用 **官方 CLI、系统命令、包管理器一键装** 是否可直接完成任务（避免从零写大段胶水代码）。
  3. **降本路径**：若仍不足，评估 **一键 shell 封装、Makefile 目标、调用小模型只做分类/提取、复用 ddgs-search / multi-search-engine + web_fetch** 等，并在任务单 `capability_gap` 写明推荐顺序与预估成本。
  4. **缺口结论**：判断编制是否 **足以执行**；若 **无人可接** 或 **缺 skill**：输出 **选项**（给哪一 agent 加何 skill、是否新 agent），**不得私自扩建**，提交 **主人审批**。
- **枢衡**：在进阶段 5 前确认绳墨已完成上述检索链摘要；可汇总绳墨结论一并报主人。

## 阶段 5 — 执行（匠作 / 丹青 / 镜语）

- 被指派的 agent 各自执行，更新任务单状态；阻塞时写明原因 @枢衡。

## 阶段 6 — 测试 / 验收（司驳）

- **司驳**：对交付做 **测试与门禁**（功能、安全、合规、滥用风险）；**准奏** 或 **封驳**（封驳须写清如何返工）。

## 阶段 7 — 回奏（枢衡）

- **司驳** 准奏后，**枢衡** 向主人做 **最终汇报**：结果摘要、残留风险、后续建议。

### 🔹 阶段1：题材筛选与立项
- 触发：主人提出项目需求（如：想做都市异能题材）或指定题材方向
- 责任人：枢衡（统筹）+ 计府（市场分析）
- **产出流程：**
  1. **市场调研**：搜索当前热门网络小说/短剧题材，筛选3-5个符合要求的题材方向
  2. **大纲撰写**：每个题材由剧绎产出「小说大纲+大概剧情发展方向」，包含设计亮点、市场价值、投入成本、风险点
  3. **主人筛选**：将所有大纲方案提交主人审批，主人选择具体创作哪个小说
  4. **正式立项**：主人选择后，按照规范创建项目空间，输出项目可行性分析报告
- **强制规则：**
  1. 每个题材必须包含：小说大纲（500-1000字）、剧情发展脉络、设计亮点、市场价值评估、成本预估、风险点
  2. 立项审批通过后，立即在`/Users/graves/Workspace/film-projects/`下创建对应项目独立空间，规范命名为`{项目名称}`（如：外卖员的时间暂停），创建时间自动记录到项目README.md中
  3. 立项后自动激活「自我提升技能」，全项目周期持续积累创作经验
- **阶段1交付物**：小说大纲×N、剧情发展方向×N、项目可行性报告（主人审批后）

### 🔹 阶段2：小说创作（双Agent模式）- 必须遵循4步生成流程
- **责任人：剧绎（第一产出）+ 镜语（审核）+ 枢衡（监督）**
- **剧绎职责（强制规范）**：
  1. **必须先完成完整大纲设计**：包括人物设定（5人以上详细设定）、世界观设定（能力体系、组织势力）、主线剧情起承转合、36章详细章节纲
  2. **章节纲模板**：每章必须包含【字数目标800-1500字、场景地点、出场人物、核心事件、剧情目标、本章爽点、铺垫/伏笔、与上章关联、结局悬念】
  3. **分批生成规则**：由于上下文限制，单次生成量有限，剧绎必须分6批生成（每批6章），每批生成后需自检完整性
  4. **铺垫/埋坑机制**：前期章节必须埋下3-5个关键伏笔，中后期章节必须回收伏笔，形成完整闭环
  5. **禁止跳过步骤**：未完成大纲设计绝对不能开始写正文！
- **镜语职责（严格审核标准，拥有一票否决权）**：
  1. **生成前审核**：审核36章章节纲是否完整、是否包含所有必需字段
  2. **生成中审核**：每批次完成后，核对：
     - [ ] 字数是否达标（每章800-1500字）
     - [ ] 章节是否完整（不能缺失章节）
     - [ ] 剧情是否偏离章节纲
     - [ ] 上下文是否连贯（与前后章节关联）
     - [ ] 伏笔是否埋下（前期章节必须埋）
  3. **质量问题判定（可驳回）**：
     - 章节丢失：任一章节缺失 → 驳回重做
     - 剧情偏离：与章节纲严重不符 → 驳回重做
     - 上下文错误：前后情节矛盾 → 驳回重做
     - 内容太短：单章少于500字 → 驳回重做
     - 节奏拖沓/爽点不足：整体质量不达标 → 驳回重做
  4. **通过标准**：只有完整、连贯、质量达标的章节才能进入3轮迭代优化流程
- **流程（4步强制执行）**：
  1. **第1步：设计完整大纲** → 剧绎输出36章章节纲 → 镜语审核 → 主人审批
  2. **第2步：分批生成章节** → 剧绎分6批生成36章 → 每批自检 → 镜语审核
  3. **第3步：3轮迭代优化** → 镜语审核通过后才能进入迭代
  4. **第4步：整合润色** → 全文通读 → 镜语最终签字
- **迭代机制（强制执行）**：
  1. 必须完成满3轮「剧绎输出→镜语审核→优化迭代」循环，不得少于3轮
  2. 每15分钟自动同步一次迭代进度到项目群
  3. 所有创作过程版本、修改意见、决策逻辑必须100%沉淀到项目空间
- **验收与交付**：
  1. 3轮迭代完成后，由镜语签字确认内容达标
  2. 输出完整小说（36章×800-1500字=约36,000字）提交主人审批
  3. 主人审批通过后，进入阶段3
- **阶段2交付物（新增分块规范，适配上下文限制）**：
  - 基础交付物：
    - 36章完整章节纲（通过审批）
    - 镜语审核记录（每批+每轮迭代）
    - 迭代记录（3轮）
  - 小说交付物（分三种形式，适配不同场景）：
    1. **完整全量版**：单文件包含36章完整正文，用于存档
    2. **分卷拆分版**：拆为6卷，每卷6章，每卷约7000字，刚好适配大模型上下文限制，后续改编/生成提示词时按需读取对应卷，避免上下文浪费
    3. **核心设定摘要版**：1000字以内，包含人物设定、世界观、主线大纲，供后续环节快速对齐核心设定，无需读全文
- **后续阶段使用规则（强制执行）**：
  1. 所有环节优先读取核心设定摘要，对齐人设和世界观
  2. 改编单集/单章内容时，仅读取对应分卷内容，禁止加载全文浪费上下文
  3. 全文校验/通读场景才允许读取完整全量版

### 🔹 阶段3：小说改编剧本（双Agent模式）
- **责任人：剧绎（改编）+ 镜语（审核）**
- **产出要求（含新优化强制规范）：**
  1. 目录规范：项目空间下必须创建4个独立目录分类存放内容，不得混合归档：

     - 必须单独产出「人物设定归档表」：包含所有出场人物的统一形象描述、能力特征、习惯/口头禅，确保视频生成时前后人物一致

  3. 第一版输出要求：必须产出**全量完整创作资产**，包含全季完整小说、所有集数的详细剧本、全部拍摄/制作脚本、全部分镜设计、提示词包，风格要求贴合真实网络用语，符合年轻用户喜好，不得仅输出部分内容
  4. 完成后提交给镜语做内容质量审核
- **协同审核责任人：镜语**
  1. 对全量内容的逻辑、叙事、风格、分镜合理性、制作可行性提出完整修改意见
  2. **新增审核点（新优化要求）**：镜语必须重点检查：
     - [ ] 人物形象前后无矛盾
     - [ ] 场景设定无冲突
     - [ ] 提示词严格引用归档描述
  3. 剧绎根据意见逐点迭代优化后重新提交给镜语
- **迭代机制（强制执行）：**
  1. 必须完成满3轮「剧绎输出→镜语审核→优化迭代」循环，不得少于3轮
  2. 每15分钟自动同步一次迭代进度到项目群，更新当前迭代轮次、修改内容、剩余待优化点
  3. 所有创作过程版本、修改意见、决策逻辑、迭代历史必须100%沉淀到项目空间，不得遗漏
- **验收与交付要求：**
  1. 3轮迭代完成后，由镜语签字确认内容达标
  2. 最终交付物：小说大纲、剧本大纲、前3集详细剧本（供主人审批）+ 全季完整剧本/分镜/制作脚本/人物归档/场景库（归档）
  3. 无论主人审批是否通过，所有内容同步更新到组织技能库，方便后续同类项目复用与自我提升
- **阶段3交付物**：完整小说、完整分集剧本、剧本大纲、制作脚本、分镜设计、提示词包、人物设定归档表、常用场景库（以上均为全量）、镜语审核意见、迭代记录、主人审批签字

## 📋 项目进展追踪与异常处理规范

### 一、追踪文件结构

每个项目目录下必须创建以下追踪文件：

```
{项目名称}/
├── README.md              # 项目基础信息 + 创建时间
├── progress.md            # 主进度台账（含实时进展检查点）
├── issues.md              # 问题记录 + 异常记录
├── workflow.log           # 工作流转日志（谁在什么时候做了什么）
├── participants.md        # 参与人员及分工
├── recovery/              # 恢复点存档（每次检查点自动备份）
└── 阶段X/                 # 各阶段交付物归档
```

### 二、进度追踪规范

#### 1. progress.md - 主进度台账

每个项目必须包含：
```markdown
# {项目名称} 进度台账
创建时间：YYYY-MM-DD
当前阶段：阶段X - 任务名称

| 阶段 | 状态 | 责任人 | 开始时间 | 结束时间 | 完成度 | 当前进展 |
|-----|-----|-------|---------|---------|-------|---------|
| 阶段1 | ✅ 完成 | 枢衡/计府 | MM-DD HH:MM | MM-DD HH:MM | 100% | 主人已选定题材 |
| 阶段2 | 🔄 进行中 | 剧绎/镜语 | MM-DD HH:MM | - | 30% | 第1轮镜语审核中 |

## 实时进展 [HH:MM 更新]
当前任务：{任务名称}
当前阶段：阶段X - 第N轮迭代
总耗时：X分钟 / 预计X分钟
完成度：X%（约完成第X步，共X步）
当前执行人：{角色}
正在做：{具体动作}
下一步：{下一步动作}
```

#### 2. 每15分钟更新机制

- 任务执行期间，每15分钟自动更新一次progress.md的「实时进展」部分
- 主人可通过询问枢衡或直接查看progress.md了解当前进展

### 三、异常处理规范

#### 1. 异常分类与处理策略

| 异常类型 | 常见场景 | 处理策略 | 升级阈值 |
|---------|---------|---------|---------|
| 工具/技术失败 | 网络断开、API超时、文件写入失败 | 立即重试3次，每次间隔2分钟 | 3次失败后上报主人 |
| 内容不达标 | 镜语审核连续2轮给出大量修改意见 | 回到上一个检查点重做 | 超过3轮迭代仍不达标 |
| 外部阻断 | 主人未审批、外部依赖不可用 | 暂停并通知主人 | 暂停超过24小时提醒 |
| 人员不可用 | 对应Agent无法响应 | 自动切换到备选Agent或通知主人 | 超过2小时不可用 |

#### 2. issues.md - 问题与异常记录

```markdown
# 问题记录

## 异常记录

| 编号 | 时间 | 异常类型 | 发生环节 | 问题描述 | 影响范围 | 处理动作 | 结果 | 重试次数 |
|-----|-----|---------|---------|---------|---------|---------|-----|---------|
| 01 | MM-DD HH:MM | 工具失败 | 剧绎写小说 | 写文件超时 | 丢失第2章 | 恢复到检查点 | 成功恢复 | 2次 |
```

#### 3. 恢复点机制

- 每个「交付动作」完成时自动创建恢复点，包含已完成的文件和当前进度
- recovery/目录存档每次检查点时的完整项目状态
- 异常发生时从最近的恢复点重新开始

#### 4. 中间产物处理规则

| 情况 | 处理方式 |
|-----|---------|
| 刚完成一小部分就崩溃 | 删除该部分，从头开始 |
| 完成大部分发现方向错误 | 删除全部，从恢复点开始 |
| 审核失败 | 保留当前版本作为参考，从上一版修改 |
| 外部工具崩溃 | 保留已生成文件，检查点重试 |
